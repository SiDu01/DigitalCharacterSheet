# Data and Rules

The app uses SQLite for both reference data and user-owned character data. Runtime builds do not import raw JSON files; reference data is prepared ahead of time in a seed database.

## Data Sources

Development input:

- local `5e Tools\data` folder

Generated runtime input:

- `Resources\Raw\seed\digital-character-sheet.db3`

The seed database is generated by:

```powershell
dotnet run --project Tools\SeedDatabaseBuilder\SeedDatabaseBuilder.csproj
```

If importer logic, source JSON mappings, or reference-data schema changes, rebuild the seed database.

## Reference Data

Reference data is imported from `5e Tools` into SQLite tables and represented in the app by definition models.

Major definition groups:

- spells
- spell tags
- classes
- subclasses
- spellcasting progressions
- class spell access rules
- subclass spell access rules
- races / ancestries
- subraces / race versions
- backgrounds
- feats
- items
- item groups
- magic item variants
- sources

Reference data should be treated as replaceable. User-owned character data should not depend on raw JSON being available at runtime.

## User-Owned Character Data

Character data includes:

- character base row
- classes and subclass selections
- feats
- base ability scores
- race/background/feat choice JSON
- known/prepared spells
- spell slots
- inventory
- saving throws
- skills
- tools
- languages
- fighting styles
- hidden features
- granted effects

This data must be preserved across migrations and future reference-data refresh flows.

## Stored Inputs vs Derived Values

The app should store user intent, not stale sheet results.

Stored inputs:

- selected definition IDs
- source/name fallback fields
- base ability scores
- selected choices
- manually selected proficiencies
- inventory state
- user notes and explicit overrides

Derived values:

- final ability scores after option bonuses
- skill and save modifiers
- class/subclass feature lists
- spell access
- spell slot progression
- item effects
- combat summaries
- inferred actions from feature text

Derived values may be cached locally inside a page or view model, but they should not become the canonical character state unless they represent an intentional user override.

## Choice JSON

Choice JSON stores selections that come from rules and definition data. It is currently used for:

- race choices
- background choices
- feat choices
- ability choices
- defense choices
- size choices where applicable

Important services:

- `CharacterAbilityChoiceService`
- `CharacterDefenseChoiceService`

When a definition is changed, invalid choice keys should be pruned and the remaining choices should be recalculated against the active rule requirements.

## Granted Effects

Granted effects represent rule outcomes that affect the character. Examples:

- saving throw proficiency
- skill proficiency
- skill expertise
- half proficiency
- tool proficiency
- language proficiency
- fighting style

Automatic option granted effects are rebuilt by `AppDatabase` from selected references and choice JSON. Manual selections should remain explicit and should not be overwritten by automatic recalculation.

## Items and Effects

Items are imported as definitions. Character inventory stores per-character state:

- item definition ID
- custom name/description
- quantity
- equipped
- attuned
- carried
- container
- notes
- charges

Item effects are partially supported and should continue moving toward the same dynamic effect model used by class/race/background/feat-derived effects.

## Database Versioning

The current app database version is defined in `Services\AppDatabase.cs` as `DatabaseVersion`.

Migrations live in `Services\AppDatabase.Migrations.cs`.

When adding a schema change:

1. Increase `DatabaseVersion`.
2. Add a matching `case` in `ApplyMigrationAsync`.
3. Keep the migration idempotent where practical.
4. Preserve user-owned tables and rows.
5. Rebuild the seed database if reference-data schema or importer behavior changed.

Reference-data version keys are composed into `SourceDataVersion`. If an import version changes, runtime builds expect a fresh seed database.

## Seed Database Boundary

The seed database should contain reference data. Installed app databases contain both copied reference data and user data.

Future reference-data update work should refresh reference tables from a new seed database while preserving character-owned tables. Do not solve this by importing raw JSON at runtime unless the packaging strategy changes.
